# PRB-002: 恶意扫描骚扰与日志污染分析

**日期**: 2026-02-10  
**状态**: [OPEN]  
**组件**: Security / Gateway  
**相关**: `backend.log`

---

## 1. Description (Problem)

**现象**:
服务器日志（`backend.log`）中充斥着大量来自外部IP的恶意扫描请求。这些请求主要针对常见的敏感路径（如 `/bins/`, `/backup/`, `phpMyAdmin` 等）进行探测，导致服务器此时虽然负载不高，但日志被严重污染，且存在潜在的安全风险。

**观察**:
```text
WARNING:  Invalid HTTP request received.
INFO:     204.76.203.18:55074 - "GET /bins/ HTTP/1.1" 401 Unauthorized
INFO:     204.76.203.18:34550 - "GET /bin/ HTTP/1.1" 401 Unauthorized
INFO:     204.76.203.18:51636 - "GET /backup/ HTTP/1.1" 401 Unauthorized
INFO:     89.248.164.165:45108 - "GET http%3A//ossweb-img.qq.com/... 401 Unauthorized
```
这些探测请求虽然目前被 401/404 拦截，但攻击者不仅消耗了连接资源，其扫描行为也干扰了正常的运维监控。

---

## 2. Root Cause Analysis (RCA)

**公网暴露**:
服务器端口直接暴露在公网，且未在前置网关（如 Nginx）层做严格的路径白名单或 IP 过滤。

**自动化脚本**:
攻击者使用自动化脚本对全网 IP 段进行无差别扫描，寻找未修补的漏洞或弱口令配置。由于服务器对所有请求都如实返回状态码（401/404），攻击者可以根据状态码快速判断服务器存活状态及路径是否存在。

**缺乏主动防御**:
当前系统仅被动拒绝非法请求，缺乏主动干扰或诱捕机制（Honeypot），使得攻击者可以低成本地持续骚扰。

---

## 3. Resolution (Proposed)

**策略建议**:
不再单纯被动拦截，而是实施“混沌邪恶”的主动防御策略——**幽灵陷阱（Ghost Trap）**。

**核心思想**:
对所有非法的探测请求，与其返回 401/404 告诉攻击者“这里禁止访问”，不如返回 **假的 200 OK**，并附带虚假的、无意义的数据。这不仅能通过混淆视听来浪费攻击者的分析资源，还能满足管理员“戏耍攻击者”的心理需求。

详见 `SUG-003-幽灵陷阱`。
