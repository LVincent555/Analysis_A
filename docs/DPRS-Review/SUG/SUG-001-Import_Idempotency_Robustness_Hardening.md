# SUG-001: 导入幂等性与鲁棒性加固建议

**日期**: 2026-02-09  
**状态**: [OPEN]  
**类型**: Infra  
**层级**: 模块-import  
**关联**: RES-001-Import_Duplicate_Key_Fix

---

## 选项 A：最小修复（低风险快收敛）

- **做法**：
  - 将冲突处理从“行内捕获”调整为“提交阶段统一捕获并分类”。
  - 将 `rolled_back` 纳入失败统计，避免摘要显示“失败=0”的误导。
  - 对 `in_progress` 超时状态增加自动收敛策略（超时后转 `failed` 或 `rolled_back`）。
- **收益**：
  - 立刻修正观测偏差，失败可见。
  - 解决“最新一天反复导入失败但状态不稳定”的主要运维痛点。
  - 改动面小，回归成本低。
- **风险**：
  - 仍是“按日期先删后导”的覆盖式导入，不是严格冲突级幂等。
  - 并发导入同日期时仍有竞争窗口。
- **验证**：
  - 同一日期重复执行 3 次，预期不报唯一键冲突。
  - 人为制造异常后，摘要失败数与日志失败数一致。
  - 中断进程后重启，`in_progress` 能在阈值内自动收敛。

## 选项 B：中期稳态（数据库级幂等）

- **做法**：
  - 用数据库 UPSERT（`ON CONFLICT (stock_code, date) DO UPDATE/DO NOTHING`）替代“先删后导”。
  - 状态文件仅作为观测与重试协调，不作为唯一真相源。
  - 增加“导入批次 ID + 最终一致性校验”。
- **收益**：
  - 真正冲突级幂等，可安全重复执行与断点重试。
  - 对状态文件漂移、部分重跑、网络抖动更稳健。
- **风险**：
  - 实现复杂度和测试成本明显上升。
  - 需要明确“冲突时更新字段策略”（全量覆盖/部分更新）。
- **验证**：
  - 同日多次重放、并发双进程导入、异常中断恢复均通过。
  - 导入后行数、哈希、关键指标一致。

## 选项 C（可选）：维持现状

- **做法**：继续使用当前“状态文件 + 按日期删除再导入”策略
- **收益**：无额外开发成本
- **风险**：最新日期重复导入仍可能出现冲突和状态误报；失败定位成本持续偏高
- **验证**：N/A

---

## 推荐

选项 A 先落地，随后按里程碑演进到选项 B。

**理由**：
- 当前最突出的问题是“可观测性失真 + 提交阶段错误处理不稳”，A 能快速止血。
- B 是长期正确方向，但一次性改造成本高，建议在 A 稳定后迭代。

---

## 前置条件 / 依赖

- 明确导入任务是否允许“同日期并发执行”。
- 明确冲突时字段策略（保留旧值/覆盖新值/按时间戳更新）。
- 补齐最小回归用例：重复导入、异常回滚、状态恢复、摘要一致性。
