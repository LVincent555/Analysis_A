# RES-037 导入后缓存刷新与外部板块同步卡死修复结果

## 状态

[SOLVED / UNIT-VERIFIED / INTEGRATION-BLOCKED-BY-DB]

## 背景

本次处理两个运行期问题：

1. 市场数据导入成功后，后端运行态缓存未及时刷新，导致前端仍看到旧数据。
2. 外部板块导入/同步任务可能长期卡住，状态一直停留在同步中，无法正常恢复。

## 原因定位

### 导入后缓存刷新

`MarketDataAdminAdapter._do_import_task()` 在导入完成后会调用运行态缓存重载，但原有路径会执行完整预加载流程，导入状态长时间停在“重载缓存”阶段；一旦刷新耗时或失败，前端很容易表现为数据不更新或任务未结束。

### 外部板块同步卡死

`BoardHeatManagementAdapter` 通过子进程执行 `scripts/sync_ext_boards.py`。旧实现主线程阻塞读取 stdout，并等待子进程自然退出；如果外部网络、代理、IP 池或目标服务请求挂住且子进程不输出日志，后端状态会长期保持 `is_syncing=True`。

## 修复内容

### 运行态缓存刷新异步化

修改文件：

- `backend/app/contexts/market_data/infrastructure/data_admin.py`

落地内容：

1. 新增 `import_status.cache_refresh`，单独记录缓存刷新状态。
2. 新增 `_schedule_runtime_cache_reload()`，将运行态缓存刷新放入后台 daemon thread。
3. 刷新流程改为：
   - 清理 API cache；
   - 清理 HotSpots cache；
   - 调用 `numpy_cache.reload(days=NUMPY_PRELOAD_DAYS)`；
   - 预热热点缓存。
4. `get_import_status()` 返回 `cache_refresh`，方便前端区分“导入完成”和“缓存仍在刷新”。

### 外部板块同步任务加超时与回收

修改文件：

- `backend/app/contexts/board_heat/infrastructure/management.py`

落地内容：

1. 新增任务超时配置：
   - `EXT_BOARD_TASK_TIMEOUT_SECONDS`
   - 默认 `1800` 秒。
2. 子进程 stdout 读取改为独立 daemon thread。
3. 主执行线程定期轮询：
   - 用户取消；
   - 任务超时；
   - 子进程自然退出。
4. 超时或取消时先 `terminate()`，超时未退出再 `kill()`。
5. finally 中统一复位：
   - `is_syncing=False`
   - `is_running=False`
   - `process=None`
   - `cancel_requested=False`
