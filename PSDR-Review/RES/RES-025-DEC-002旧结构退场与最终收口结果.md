# RES-025: DEC-002 旧结构退场与最终收口结果

**日期**: 2026-07-08

**层级**: Backend
**关联**: DEC-002, SUG-003, RES-024

---

## 结果摘要

已完成 DEC-002 Level 5 的本轮退场：`main.py` composition root 不再通过旧 `app.routers` 聚合装配已迁移业务路由，而是直接装配 `contexts/*/api` router。

旧 `app.routers.*` 文件保留为兼容入口，主要用于历史 import 兼容和安全网关等尚未迁入 context 的平台边界。旧 `app.services.*` 中的算法、缓存、导入脚本调用也没有被强行删除，而是作为 legacy adapter 的被包装实现继续存在。

---

## 已完成改动

1. `backend/app/main.py` 直接装配 `contexts/*/api` router。
2. 已迁移业务的旧 `app.routers.*` 文件退为兼容壳。
3. `app.routers.secure` 暂保留为平台加密网关入口。
4. `db_models.py` 暂不物理拆分。当前已建立 ownership 规则，后续若要拆 ORM 文件，应单独处理 Alembic/metadata/import 稳定性。

## 退场边界

DEC-002 是架构设计决策，不是“删除所有历史代码”的无限期任务。本次退场线定义为：

1. 新增业务默认进入 `contexts/`。
2. composition root 直接装配 context router。
3. 旧 router 只作为兼容壳，不再承载新增业务编排。
4. 旧 service 中仍有价值的算法和脚本集成由 infrastructure adapter 包装。
5. 更大范围的 ORM 拆分、任务队列化、脚本重写、secure gateway 平台化，需要另立 PRB/SUG/DEC。

---

## 结论

DEC-002 的 Level 0 到 Level 5 已完成，SUG-003 的架构方案已实施。后续不再围绕是否采用“模块化单体 + 六边形架构 + 局部 DDD + CQRS 查询侧”重复决策，新增能力按该结构直接落地。
