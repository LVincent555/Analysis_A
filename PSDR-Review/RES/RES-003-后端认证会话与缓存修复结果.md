## 执行摘要

本次执行 `SUG-001` 的后端认证会话修复，目标是解决登录有效期配置后仍被约半小时强制下线的问题，并收敛前次探查中发现的认证、会话、缓存边界风险。

已完成改动：

1. `backend/app/main.py`
   - 新增独立缓存分区 `session_keys`，用于保存 `/api/secure` 解密所需的运行时会话密钥。
   - 保留 `sessions` 分区给会话心跳/状态同步使用，避免 `session_key` 数据进入 Write-Behind 同步链路。

2. `backend/app/auth/dependencies.py`
   - 会话密钥改为按 `user_id:device_id` 存取，避免多设备登录互相覆盖。
   - `get_current_user()` 增加账号软删除、禁用、过期、锁定、token version、数据库会话存在性、会话撤销和会话过期校验。
   - 当前认证 payload 和 session 绑定到 `user._auth_payload` / `user._auth_session`，供后续依赖读取当前设备。
   - `restore_session_key_from_db()` 改为按设备查询未撤销、未过期会话；由于现有 `session_key_encrypted` 仍由密码派生密钥加密，后端重启后仍要求重新登录。

3. `backend/app/auth/jwt_handler.py`
   - Access Token 和 Refresh Token 增加可选 `ver` 字段，支持 `User.token_version` 强制失效机制。

4. `backend/app/routers/auth.py`
   - 登录时将 `UserSession.expires_at` 改为 refresh token 生命周期，不再使用 access token 生命周期。
   - 登录响应中的 `expires_at` 仍表示 access token 过期时间。
   - Refresh 时不再把数据库会话过期时间改短为 access token 过期时间，只更新 `last_active`。
   - Refresh 增加用户软删除、禁用、过期、锁定、token version、会话撤销和会话过期校验。
   - Logout 改为撤销当前设备会话，并只清理当前设备的 `session_key`。
   - 同设备重复登录不再触发设备上限淘汰；新设备超限时只淘汰最旧的有效会话。
   - `/api/auth/sessions` 只返回未撤销、未过期会话。

5. `backend/app/routers/cache_mgmt.py`
   - `/api/cache/*` 增加路由级 `require_admin` 依赖，普通用户不能直接调用缓存管理接口。

## 遗留风险 / 后续行动

1. 旧 token 兼容策略：本次仅在新签发 token 中写入 `ver`。旧 token 不含 `ver` 时不会触发 token version 比对，但仍会受到数据库会话存在性、撤销和过期校验约束。
2. 后端重启后 `session_key` 仍不能从数据库自动恢复，因为现有 `session_key_encrypted` 使用密码派生密钥加密，后端没有用户明文密码；预期行为是要求重新登录。
3. 直接 `/api/*` 请求在 `FORCE_SECURE_API=false` 且路由未显式依赖 `get_current_user` 时，仍主要依赖中间件 token 校验。安全边界更完整的方案应继续推动路由依赖统一化。
4. 建议在安装 `backend/requirements.txt` 后补跑最小接口回归：login、refresh、logout、logout-all、双设备登录、管理员撤销会话、普通用户访问 `/api/cache/*`。
