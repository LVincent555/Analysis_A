## 结论摘要

当前后端自研缓存系统没有在 DEC-002 重构中被破坏。`core/caching` 仍保留为平台层能力，`/api/cache/*` 管理入口已经迁入 `contexts.operations`，并通过 adapter 调用原有缓存能力。

建议继续使用这套缓存，但将其定位为 **项目级单体本机缓存平台**，通过登记、观测、容量控制、策略语义修正和历史缓存收口来演进。当前不建议替换为 Redis，也不建议推倒重写。

推荐路线：

1. 第一阶段先做缓存治理登记和测试基线回收。
2. 第二阶段加固 `ObjectStore`、策略语义、metrics 和管理 API。
3. 第三阶段收口 `numpy_cache`、`HotSpotsCache` 与 API 响应缓存键治理。
4. 第四阶段仅在多 worker / 多实例部署前，评估外部化安全态缓存和 nonce。

当前 Phase 0 至 Phase 5 已完成，其中 Phase 5 以实验功能形式落地。结果见 RES-026、RES-027、RES-028、RES-029、RES-032、RES-033。

## 当前事实

### 1. 现有缓存内核仍在

`core/caching` 当前仍由以下核心对象组成：

```text
CacheEntry
CachePolicy
WriteBehindPolicy
WriteThroughPolicy
CacheAsidePolicy
ObjectStore
VectorStore
FileStore
UnifiedCache
PublicCache
DatabaseSyncer
```

这套设计的基本方向是合理的：

1. `UnifiedCache` 负责 region 注册与统一访问。
2. `ObjectStore` 负责内存对象缓存。
3. `FileStore` 封装 `diskcache`，用于 API 响应和报表文件缓存。
4. `VectorStore` 原本计划封装 Numpy 类高密度查询缓存。
5. `PublicCache` 提供业务侧门面与 `KeyBuilder`。

### 2. 启动时注册的 region

`main.py::init_cache_system()` 当前注册：

| region | store | policy / 配置 | 主要用途 |
|------|------|------|------|
| `sessions` | `ObjectStore` | `WriteBehindPolicy(ttl=0, sync_interval=10)` | 会话心跳状态 |
| `session_keys` | `ObjectStore` | `WriteThroughPolicy(ttl=0)` | `/api/secure` 运行时 session key |
| `users` | `ObjectStore` | `CacheAsidePolicy(ttl=3600)` | 用户信息缓存 |
| `config` | `ObjectStore` | `WriteThroughPolicy(ttl=0)` | 系统配置缓存 |
| `api_response` | `FileStore` | 200MB diskcache | API 响应缓存 |
| `reports` | `FileStore` | 500MB diskcache | 报表文件缓存 |

手动初始化验证：

```text
['sessions', 'session_keys', 'users', 'config', 'api_response', 'reports']
```

### 3. 管理入口已经迁入 Operations

旧 `routers/cache_mgmt.py` 已经退为兼容入口。当前实际入口是：

```text
contexts.operations.api.cache_router
```

它通过 `PlatformCacheManagementAdapter` 统一调用：

1. `numpy_cache.get_memory_stats()`
2. `HotSpotsCache.get_cache_stats()`
3. `unified_cache.stats()`
4. `unified_cache.clear_api_cache()`
5. `unified_cache.clear_report_cache()`
6. `preload_cache()`
7. `unified_cache.gc()`

也就是说，管理面已经按 DEC-002 迁入 Operations，但缓存内核仍保留在 `core/caching` 平台层。

### 4. 历史缓存没有完全纳入 UnifiedCache

当前 `numpy_cache` 与 `HotSpotsCache` 仍是独立缓存体系：

1. `numpy_cache` 是 `NumpyCacheMiddleware` 单例，承载股票、板块、策略查询等大量 read-heavy 能力。
2. `HotSpotsCache` 内部使用类变量 `_cache` 保存热点榜数据。
3. `VectorStore(stock_market)` 在 `core/caching` 中存在，但启动时未注册 `stock_market` region。
4. 多个 context infrastructure 仍直接依赖 `numpy_cache`，这是当前架构允许的 legacy adapter 边界，但还没有统一治理。

### 5. 缓存内核老测试不在默认基线

`backend/pyproject.toml` 当前只收集：

```text
testpaths = ["tests/unit"]
```

因此 `backend/tests/test_caching.py` 不在默认 `uv run pytest` 基线里。

本次手动运行结果：

```text

```

结果出现 1 个失败：


原因不是生产代码损坏，而是该老测试仍把 `config` region 注册为 `CacheAsidePolicy(ttl=0)`；真实启动环境已经改为 `WriteThroughPolicy(ttl=0)`。这说明测试事实已经落后于当前实现，需要迁入并校准。

当前默认相关测试仍通过：


## 主要问题

### 1. 缺少 region 登记与所有权表

目前 region 事实散落在 `main.py`、`facade.py`、Operations adapter 和具体业务调用中。后续维护者很难快速回答：

1. 每个 region 谁拥有。
2. key 如何命名。
3. TTL 是多少。
4. 是否允许丢失。
5. 是否允许脏读。
6. 何时失效。
7. 是否能跨进程一致。

### 2. ObjectStore 无容量上限

`ObjectStore` 当前有 TTL 与 `clear_expired()`，但没有 `max_entries`、LRU 淘汰或内存水位保护。

风险：

1. `ttl=0` 的 region 永不过期。
2. key 异常增长时没有硬保护。
3. `sessions`、`session_keys`、`config` 这类长期 region 需要明确上限。

### 3. 策略语义存在可加固点

具体观察：

1. `WriteThroughPolicy.set()` 先写内存再执行 `persister`，如果 `persister` 失败，缓存可能保留未落库值。
2. `CacheAsidePolicy.get()` 捕获 loader 异常后直接返回 `None`，可用性好，但故障容易被吞掉。
3. `WriteBehindPolicy.get_dirty_keys()` 获取脏 key 时直接清空集合；默认 session 同步路径如果 DB 写入失败，dirty key 可能丢失重试机会。
4. `safe_cache_call` 保护业务不被缓存异常拖垮，但没有把错误计入 metrics，容易静默退化。

### 4. 可观测性不足

当前 stats 主要是 region 级数量、过期数、dirty 数、磁盘大小等基础信息。

缺少：

1. hit / miss。
2. set / delete。
3. loader_error / persister_error。
4. eviction / expired_cleanup。
5. FileStore recovery 次数。
6. WriteBehind sync 成功/失败次数。
7. region 级容量水位。

### 5. 管理 API 能力偏粗

当前 `/api/cache/*` 能看到 `numpy_cache`、`hotspots_cache` 与 `unified_cache` 的基本统计，也能清理 hotspots/api/report。

不足：

1. 不能按 UnifiedCache region 精确清理。
2. 不能看到每个 region 的所有权、TTL、策略、容量上限。
3. 不能区分 health degraded 与 healthy。
4. 不能展示 WriteBehind 脏数据积压、sync 失败或 FileStore recovery。

### 6. `numpy_cache` 与 `HotSpotsCache` 治理不统一

当前它们可以被 Operations 管理面看到，但没有被注册为 `UnifiedCache` 的正式 region。

这并非短期 bug，但会带来认知成本：

1. 有些缓存通过 `cache` 门面。
2. 有些缓存通过 `numpy_cache` 单例。
3. 有些缓存通过 `HotSpotsCache` 类变量。
4. 热点榜又同时会预加载进 `api_response` FileStore。

### 7. 单进程假设需要写清楚

当前缓存体系是单进程内存 + 本机磁盘缓存，适合本项目当前单体部署。

如果未来使用多 worker、多进程或多实例：

1. `session_keys` 会按进程分裂。
2. `login_nonce_store` / `secure_nonce_store` 会按进程分裂。
3. `config` 缓存会按进程分裂。
4. `FileStore` 本机磁盘缓存也会按实例分裂。

本项目目前可以接受单进程假设，但必须在方案里把边界写明。

## 选项 A：维持现状，只补文档

- **做法**：
  - 写 region 表。
  - 不调整代码。
  - 不迁移测试。
- **收益**：
  - 成本最低。
  - 没有行为风险。
- **风险**：
  - 只能解决“忘了怎么搞”的问题。
  - ObjectStore 无容量、metrics 不足、策略语义等问题仍存在。
  - 后续继续扩展时容易再次散开。
- **验证**：
  - 文档存在即可。

## 选项 B：轻量治理与加固，保留单体本机缓存定位

- **做法**：
  - 建立 region registry 与所有权表。
  - 把老缓存测试迁入 `tests/unit` 并按当前事实修正。
  - 增加 ObjectStore 容量上限和基础 LRU。
  - 增加统一 metrics。
  - 修正 WriteThrough / WriteBehind 的失败语义。
  - 扩展 Operations 管理 API。
  - 逐步把 `numpy_cache`、`HotSpotsCache` 纳入注册或 adapter 治理。
- **收益**：
  - 保留现有高性能路径。
  - 改动可分阶段落地。
  - 对 DEC-002 架构方向兼容。
  - 能显著提升可维护性和可观测性。
- **风险**：
  - 需要补较多单元测试。
  - 容量淘汰和策略语义变化需要小心保持兼容。
  - `numpy_cache` 体量较大，不能强行一次性改造。
- **验证**：
  - `uv run pytest` 通过。
  - 缓存内核测试进入默认基线。
  - `/api/cache/stats` 可展示 region policy、ttl、capacity、metrics。
  - 原有业务 API contract 不变。

## 选项 C：引入 Redis，重做为分布式缓存

- **做法**：
  - 增加 Redis 依赖。
  - session key、nonce、config、api cache 等迁 Redis。
  - 本机缓存只保留 Numpy read model。
- **收益**：
  - 多 worker / 多实例一致性更好。
  - 安全 nonce 和 session key 可以跨进程共享。
- **风险**：
  - 当前个人项目收益不足。
  - 部署复杂度上升。
  - 会把问题从“本机缓存治理”变成“外部基础设施治理”。
  - Numpy 分析缓存仍不能简单迁 Redis。
- **验证**：
  - 需要新增 Redis integration profile。

## 选项 D：推倒重写缓存系统

- **做法**：
  - 重写 `core/caching`。
  - 用成熟库或新抽象替换现有 `UnifiedCache`。
- **收益**：
  - 可以一次性重设架构。
- **风险**：
  - 成本最高。
  - 容易破坏分析查询性能。
  - 现有缓存系统已有可用基础，推倒重写收益不明显。
- **验证**：
  - 需要大范围回归。

---

## 推荐

推荐采用 **选项 B：轻量治理与加固，保留单体本机缓存定位**。

理由：

1. 当前项目仍是单体后端，缓存主要用于本机加速、会话运行态和分析 read model。
2. 自研缓存内核方向正确，有 region、store、policy、facade 和 syncer，不需要重写。
3. 真正欠缺的是治理、测试、容量、观测和历史缓存收口。
4. `numpy_cache` 是项目性能核心，应该适配和登记，不应贸然替换。
5. Redis 可以作为未来多进程/多实例前置条件，不是当前第一步。

## 目标架构

目标仍保持平台层缓存内核：

```text
backend/app/core/caching/
  registry.py          # region spec 与默认注册清单
  entry.py             # CacheEntry
  metrics.py           # CacheMetrics / RegionMetrics
  manager.py           # UnifiedCache
  store.py             # ObjectStore / VectorStore / FileStore
  policy.py            # CachePolicy
  policies/
  facade.py            # PublicCache + KeyBuilder
  syncer.py            # DatabaseSyncer
```

上下文通过 port / adapter 使用缓存：

```text
context/application -> CachePort
context/infrastructure -> Core cache adapter
core/caching -> store / policy / metrics
```

允许：

1. `infrastructure` 依赖 `core.caching`。
2. `operations` 通过 adapter 管理缓存。
3. read-heavy 查询侧使用 Numpy/HotSpots adapter。

禁止：

1. `domain` 依赖缓存。
2. `application` 直接操作 `UnifiedCache` 具体 region。
3. 新业务绕过 `KeyBuilder` 或 region registry 手写散乱 key。

## Region 登记建议

建议新增 `backend/app/core/caching/registry.py`：

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class CacheRegionSpec:
    name: str
    owner: str
    store_type: str
    policy: str
    ttl_seconds: int
    max_entries: int | None
    consistency: str
    disposable: bool
    clearable: bool
    description: str
```

初始登记表：

| region | owner | store | policy | ttl | max_entries 建议 | 一致性 | 可丢弃 |
|------|------|------|------|------|------|------|------|
| `sessions` | identity | ObjectStore | WriteBehind | 0 | 10000 | 最终一致 | 是 |
| `session_keys` | identity/security | ObjectStore | WriteThrough | 0 | 10000 | 进程内强一致 | 否 |
| `users` | identity | ObjectStore | CacheAside | 3600 | 10000 | 可短暂旧读 | 是 |
| `config` | operations | ObjectStore | WriteThrough | 0 | 1000 | 进程内强一致 | 可重建 |
| `api_response` | platform | FileStore | disk TTL | 默认 24h | disk 200MB | 可旧读 | 是 |
| `reports` | platform | FileStore | disk TTL | 默认 24h | disk 500MB | 可旧读 | 是 |
| `stock_market` | market_data/analysis | VectorStore | read-only | reload | N/A | 以 DB 导入数据为准 | 可重建 |
| `hot_spots` | analysis | HotSpots adapter | TTL | 25h | 10 dates | 可旧读 | 是 |

说明：

1. `stock_market` 与 `hot_spots` 可先作为 logical region 登记，不必第一步改底层实现。
2. `session_keys` 虽使用缓存接口，但不是普通性能缓存，是运行态安全材料；后续多进程前必须外部化。
3. `login_nonce_store` 与 `secure_nonce_store` 当前属于安全 replay store，可在未来单独登记为 security ephemeral store。

## 分阶段实施计划

### Phase 0：治理冻结与测试回收

目标：先让一年后的维护者能看懂、能验证。

改动：

1. 新增 `core/caching/registry.py`，登记当前 region。
2. 将 `init_cache_system()` 的 region 注册逻辑从 `main.py` 抽到 `core/caching/bootstrap.py` 或 `registry.py`。
3. `main.py` 只调用 `init_default_cache_regions(base_dir)`。
4. 把 `backend/tests/test_caching.py` 拆迁到 `backend/tests/unit/test_cache_core.py`。
5. 修正老测试里 `config` region 的策略，从 `CacheAsidePolicy` 改为 `WriteThroughPolicy`。
6. 新增启动注册测试：确认默认 region 与 registry 一致。
7. 写 `core/caching/README.md`，记录 region、key、策略、边界。

验收：

```text
uv run pytest -q tests/unit/test_cache_core.py
uv run pytest
```

并确认：

1. 缓存内核测试进入默认基线。
2. `main.py` 不再直接堆 region 细节。
3. `config` 测试符合当前生产事实。

当前状态：已完成，结果见 RES-026。

### Phase 1：ObjectStore 容量与 metrics

目标：让内存缓存有边界、有观测。

改动：

1. 新增 `CacheMetrics`：
   - `hits`
   - `misses`
   - `sets`
   - `deletes`
   - `loader_errors`
   - `persister_errors`
   - `evictions`
   - `expired_cleanups`
   - `last_error`
2. `ObjectStore` 支持：
   - `max_entries: int | None`
   - `eviction_policy="lru"`
   - set 时超过上限自动淘汰最久未访问条目。
3. `stats()` 返回：
   - policy 名称
   - ttl
   - max_entries
   - active / expired / dirty
   - metrics
4. `FileStore` 将 recovery 次数、operation errors 纳入 stats。
5. `safe_cache_call` 继续保护业务，但记录错误 metrics。

验收：

1. ObjectStore 超过 `max_entries` 会 LRU 淘汰。
2. hit/miss/set/delete 指标可测试。
3. `/api/cache/stats` 能看到 region metrics。
4. 现有 API 行为不变。

当前状态：已完成，结果见 RES-027。

### Phase 2：策略语义修正

目标：降低缓存策略的隐性一致性风险。

改动：

1. `WriteThroughPolicy.set()` 改成：
   - 有 `persister` 时先执行 persister。
   - persister 成功后写缓存。
   - persister 失败时不污染缓存，并记录 error。
2. `CacheAsidePolicy.get()`：
   - loader 异常仍可返回 `None`，但必须记录 metrics 和日志。
   - 可选增加 `raise_loader_errors` 开关，仅测试或关键 region 开启。
3. `WriteBehindPolicy` 增加：
   - `peek_dirty_keys()`
   - `ack_dirty_keys(keys)`
   - `requeue_dirty_keys(keys)`
4. `DatabaseSyncer` 改为 DB commit 成功后 ack dirty key。
5. session sync 失败时保留 dirty key，等待下次重试。

验收：

1. WriteThrough persister 失败不会写入缓存。
2. WriteBehind DB 失败后 dirty key 保留。
3. force_sync 后成功清理 dirty key。
4. 策略单测覆盖失败路径。

当前状态：已完成，结果见 RES-028。

### Phase 3：管理 API 增强

目标：让后台能看见缓存健康，而不是只看到粗略 stats。

改动：

1. `/api/cache/stats` 返回 registry 信息：
   - owner
   - store_type
   - policy
   - ttl
   - max_entries
   - disposable
   - clearable
2. `/api/cache/health` 区分：
   - `healthy`
   - `degraded`
   - `error`
3. `ClearCacheUseCase` 支持：
   - `cache_type=api`
   - `cache_type=report`
   - `cache_type=hotspots`
   - `cache_type=region:<name>`
   - `cache_type=all_disposable`
4. 明确禁止清理或谨慎清理：
   - `session_keys`
   - security nonce store
5. `ReloadAllCacheUseCase` 拆分为：
   - reload numpy read model
   - reload hotspots
   - reload config
   - reload all

验收：

1. 普通用户不能访问管理 API。
2. region 级清理只允许 clearable region。
3. health 可反映 Numpy 未加载、FileStore 损坏恢复、WriteBehind 积压等状态。

当前状态：已完成，结果见 RES-029。

### Phase 4：历史缓存收口

目标：不破坏性能地把 `numpy_cache` 与 `HotSpotsCache` 纳入治理视图。

改动：

1. 注册 logical region `stock_market`：
   - 先使用 `VectorStore(numpy_cache)`。
   - 保持现有 context infrastructure 对 `numpy_cache` 的直接注入。
   - 新增代码优先通过 adapter 获取。
2. 注册 logical region `hot_spots`：
   - 可先用 adapter 包装 `HotSpotsCache.get_cache_stats()`、`clear_cache()`。
   - 暂不强行重写 `_cache` 结构。
3. `preload_cache()` 改成通过 cache platform adapter 编排：
   - reload stock_market
   - preload hot_spots
   - write api_response hotspot snapshot
4. 新增 `AnalysisCachePort`、`MarketDataCachePort`，让后续查询侧不直接 import 历史 service 单例。

验收：

1. Operations stats 能展示 `stock_market` 和 `hot_spots` logical region。
2. 原分析查询性能不明显下降。
3. 原 API contract 不变。
4. 新增查询侧默认走 cache port / adapter。

当前状态：已完成，结果见 RES-032。本次以治理视图、registry、bootstrap 与管理入口收口为主；查询侧保持既有 context adapter 注入方式，不强行重写高频 Numpy 查询路径。

### Phase 5：多进程前置方案

目标：如果后续部署形态从单进程变成多 worker / 多实例，提前有路线。

触发条件：

1. 使用 `uvicorn --workers > 1`。
2. 部署多个后端实例。
3. 需要跨进程共享登录 nonce 或 session key。

建议：

1. 不迁移 Numpy read model 到 Redis。
2. 将以下运行态外部化：
   - `session_keys`
   - `login_nonce_store`
   - `secure_nonce_store`
   - config cache version / invalidation signal
3. 已先落地实验 adapter：
   - `RuntimeStateStore`
   - `InMemoryRuntimeStateStore`
   - `DiskCacheRuntimeStateStore`
4. 后续 Redis 外部化 adapter 可命名为：
   - `RedisSessionKeyStore`
   - `RedisReplayNonceStore`
   - `RedisConfigInvalidationBus`
5. `api_response` 与 `reports` 可继续本机 FileStore，除非需要跨实例共享报表。

验收：

1. 多 worker 下 secure-login replay 防护仍有效。
2. 多 worker 下 `/api/secure` session key 不因进程切换失效。
3. 单进程模式仍不强依赖 Redis。

当前状态：已完成，结果见 RES-033。当前实现是实验 `diskcache` 后端，只适合同机多 worker；跨机器多实例仍需另签 DEC 引入 Redis 等外部运行态服务。
