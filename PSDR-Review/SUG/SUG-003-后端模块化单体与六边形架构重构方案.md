## 背景

当前后端早期参照 Java 三层结构演进，形成了 `routers/`、`services/`、`auth/`、`core/`、`db_models.py` 等横向目录。随着认证、会话、缓存、用户管理、股票分析、板块热度、同步导入和后台管理功能叠加，代码开始出现以下倾向：

1. 业务链路横跨多个横向目录，阅读和修改成本上升。
2. `router` 中仍保留较多业务规则，`service` 目录逐步变成杂物间。
3. 认证、会话、缓存、配置和审计的边界不稳定，容易互相污染。
4. 股票分析类接口以查询和计算为主，不适合强行套完整 DDD 聚合模型。
5. 后续计划继续扩展功能，项目重启阶段适合先明确架构方向。

本建议的目标不是追求形式化 DDD，而是把后端整理为可持续扩展的模块化单体。

---

## 选项 A：继续三层架构并做局部清理

- **做法**：
  - 保留 `routers/`、`services/`、`db_models.py` 结构。
  - 将过大的 router 拆小，补充 service 函数。
  - 对认证和缓存等高风险模块做局部规约。
- **收益**：
  - 改动成本低。
  - 对现有代码和 API 兼容性影响最小。
- **风险**：
  - 只能缓解表面混乱，不能解决模块边界不清的问题。
  - 后续继续加功能时，`services/` 仍会扩张成横向大目录。
  - Java 式三层在 Python/FastAPI 中容易产生过多机械转发。
- **验证**：
  - 文件数量减少或 router 变薄。
  - 但无法形成长期稳定的架构边界。

## 选项 B：完整 DDD 全量重构

- **做法**：
  - 为所有业务对象建立 Entity、Value Object、Aggregate、Repository、Domain Service、Application Service。
  - 所有读写都经过领域模型和仓储接口。
  - 全量拆分现有 ORM model 和领域对象。
- **收益**：
  - 理论边界最完整。
  - 认证、会话、权限、策略等复杂状态规则会明显受益。
- **风险**：
  - 对股票排行、趋势、板块热度这类 read-heavy 分析查询不经济。
  - 容易为了“纯 DDD”引入大量样板代码。
  - 大改周期长，容易在项目重启期拖慢功能恢复。
- **验证**：
  - 领域层可脱离 FastAPI/SQLAlchemy 单测。
  - 但查询性能和实现复杂度需要额外控制。

## 选项 C：模块化单体 + 六边形架构 + 局部 DDD + CQRS 查询侧

- **做法**：
  - 按业务上下文拆分目录，不再按 controller/service/dao 横向分层。
  - 每个上下文内部使用 `api / application / domain / infrastructure` 四层。
  - 对认证、会话、权限、配置、审计等规则密集区域采用局部 DDD。
  - 对股票分析、排行榜、趋势、热度等读密集接口采用 CQRS 查询侧，允许直接使用优化 SQL、Pandas、Numpy、缓存和读模型。
  - 保持单体部署，不拆微服务。
- **收益**：
  - 兼顾长期边界和 Python 实用性。
  - 认证等复杂规则可沉入领域层，分析查询仍保持高效直接。
  - 后续增加新功能时，可以自然落入对应上下文。
  - 比完整 DDD 少样板，比三层结构更有边界。
- **风险**：
  - 初期迁移需要建立新骨架和 import 规则。
  - 短期内新旧目录会共存，需要控制迁移顺序。
  - 团队或后续维护者需要遵守上下文边界，否则会退化成新目录下的旧混乱。
- **验证**：
  - 新功能可明确归属到一个上下文。
  - `api` 层只做 HTTP 适配，业务规则不在 router 中扩散。
  - `domain` 层不依赖 FastAPI、SQLAlchemy、缓存、JWT。
  - 查询侧可绕过领域实体，但必须被标记为 query service/read model。

## 选项 D：停止架构重构，只在新增功能时局部调整

- **做法**：维持当前结构，新功能按现有目录添加。
- **收益**：短期无重构成本。
- **风险**：
  - 当前混乱会随新增功能继续放大。
  - 后续再整理时，需要同时处理更多历史兼容和隐性耦合。
- **验证**：N/A

---

## 推荐

推荐采用 **选项 C：模块化单体 + 六边形架构 + 局部 DDD + CQRS 查询侧**。

**理由**：

1. 项目仍是单体应用，拆微服务没有收益。
2. 后端确实需要从 Java 三层转向更适合 Python 的按业务模块组织。
3. 认证、会话、权限、配置、审计有真实领域规则，适合局部 DDD。
4. 股票分析和板块热度主要是查询、计算和缓存，适合 CQRS 查询侧，不适合强塞聚合根。
5. 该方案能支持后续扩展，同时避免“完整 DDD 仪式感工程”。

## 迁移控制点

1. **API contract 基线**：每个 Level 前记录路径、方法、状态码和 JSON key；迁移后必须对比，不在架构迁移中顺手改前端可见行为。
2. **composition root 收敛**：`main.py` 后续只负责装配 router、middleware、生命周期和 adapter，不承载业务初始化细节。
3. **ORM ownership 表**：`db_models.py` 物理拆分前，先建立模型归属表；SQLAlchemy metadata 和 import 兼容稳定后再拆文件。
4. **数据库配置治理**：处理 `database.py` 被忽略的问题，目标是版本化数据库连接模块，私密连接信息只放 `.env`。
5. **测试基线**：默认使用 `cd backend; uv run pytest`，只收集离线单元测试；真实服务、真实数据和外部网络测试迁入 integration profile。
6. **import 边界检查**：Level 0 后应增加自动检查，确保 `domain` 不 import FastAPI、SQLAlchemy、JWT、cache、router、schema。
7. **兼容壳策略**：旧 `routers/`、`services/`、`auth/` 在迁移期可以保留，但只作为转发/兼容入口，不继续承载新业务。

## 分层规则

### api 层

职责：

- FastAPI router
- request/response schema
- 依赖注入
- HTTP 状态码和错误映射

禁止：

- 写复杂业务规则
- 直接操作缓存细节
- 写复杂 SQL
- 直接跨上下文访问其它上下文的 infrastructure

### application 层

职责：

- 用例编排
- command/query handler
- transaction 边界
- 调用 port
- 组合 domain 规则和 infrastructure adapter

禁止：

- 依赖 FastAPI
- 直接构造 HTTPException
- 直接依赖具体数据库实现

### domain 层

职责：

- 业务实体
- 值对象
- 领域策略
- 领域服务
- 业务不变量

禁止：

- import FastAPI
- import SQLAlchemy Session
- import Redis/diskcache/cache manager
- import JWT 库
- import router/schema

### infrastructure 层

职责：

- SQLAlchemy model 和 repository 实现
- JWT/token adapter
- cache adapter
- 外部数据源 adapter
- 文件、Excel、Akshare、Pandas/Numpy 适配

允许：

- 依赖 core
- 依赖 domain
- 实现 application ports

## CQRS 使用规则

命令侧适用：

- 登录
- 登出
- 刷新 token
- 创建用户
- 修改用户
- 撤销会话
- 导入数据
- 同步外部数据
- 更新配置
- 清理缓存

查询侧适用：

- 股票排行
- 板块趋势
- 热点列表
- 操作日志查询
- 分析结果查询
- 报表生成

查询侧规则：

1. 可以直接使用 read repository。
2. 可以返回 DTO/read model，不必构造领域实体。
3. 可以使用优化 SQL、Numpy、Pandas 和缓存。
4. 不允许把写入规则藏在 query service 中。
5. 如果查询接口产生副作用，必须改归 command。

## 前置条件 / 依赖

1. 先接受新旧目录短期共存。
2. 每阶段迁移必须保持 API 路径兼容，除非另立 DEC。
3. 重构期间不把功能大改和架构移动混在同一个提交里。
4. 每个上下文迁移完成后写 RES，记录迁移范围、验证结果和遗留项。
5. 后续新增功能默认进入 `contexts/`，不再优先写入旧 `routers/` / `services/`。
6. 每次架构迁移提交前运行 `cd backend; uv run pytest`，并记录是否涉及 integration 测试。
