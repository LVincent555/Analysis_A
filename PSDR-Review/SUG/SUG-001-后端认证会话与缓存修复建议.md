# SUG-001: 后端认证会话与缓存修复建议

**日期**: 2026-07-08
**状态**: [OPEN]
**类型**: Infra / Backend / Auth / Cache / Refactor
**层级**: Backend
**关联**: PRB-001

---

## 选项 A：最小风险修复

- **做法**：
  - 在 `auth.py` 补充 `logger = logging.getLogger(__name__)`。
  - 为 `/api/cache/*` 增加管理员依赖。
  - `logout()` 根据 JWT 中的 `device_id` 撤销或删除当前用户当前设备的 `UserSession`。
  - `refresh` 查询会话时补充 `is_revoked == False`、`expires_at > now`、用户未过期/未软删除校验。
  - 暂不重构 `sessions` 缓存分区，只先降低显性运行时错误。
- **收益**：
  - 改动范围小，回归成本低。
  - 能快速修复 refresh 500、缓存管理越权、logout 不闭环等明确问题。
- **风险**：
  - 不能根治 `sessions` 分区混用。
  - 多设备 session key 覆盖和同步器误写库风险仍存在。
- **验证**：
  - 运行 Python 语法检查。
  - 手动验证普通用户无法调用 `/api/cache/*`。
  - 手动验证 refresh 异常路径返回 401。
  - 手动验证 logout 后 refresh token 不再可用。

## 选项 B：认证会话链路完整修复

- **做法**：
  - 将运行时解密 key 缓存从 `sessions` 拆到独立分区，例如 `session_keys`。
  - `session_keys` key 使用 `user_id:device_id` 或 `session_id`，不再只按 `user_id` 存储。
  - JWT payload 增加 `session_id` 和 `token_version`，或至少在认证时通过 `user_id + device_id` 查找当前有效 `UserSession`。
  - `get_current_user()` 统一校验用户状态、账号过期、软删除、会话撤销、会话过期和 token version。
  - `DatabaseSyncer._sync_sessions()` 只同步明确的心跳分区，避免同步解密 key 数据。
  - 补齐会话撤销、logout、logout-all、change-password 的统一失效逻辑。
- **收益**：
  - 根治多设备 key 覆盖。
  - 强制下线、撤销会话、账号过期、软删除可以形成一致认证边界。
  - 缓存分区职责清晰，降低后续维护风险。
- **风险**：
  - 改动涉及登录、刷新、secure 网关、会话管理、缓存同步，回归面较大。
  - 旧 token 和旧客户端会话需要兼容策略，可能要求用户重新登录。
- **验证**：
  - 单设备登录、双设备登录、刷新、登出、全部登出、管理员撤销会话。
  - 后端重启后 secure 请求行为是否符合预期。
  - `sessions`/`session_keys` 分区 dirty keys 不再互相污染。
  - 用户禁用、软删除、账号过期后旧 token 不可继续访问。

## 选项 C：下载链路专项修复

- **做法**：
  - 操作日志 CSV 导出不走当前 JSON-only secure 网关，改为后端提供一次性下载 token 或短期 signed URL。
  - 或扩展 secure 网关支持二进制/文本响应，返回 `content_type`、`filename`、`base64`。
  - 前端 `OperationLogs.js` 不再 `window.open` 直连 `/api/admin/logs/export/csv`，改为带认证的下载流程。
- **收益**：
  - 修复强制 secure 模式下 CSV 导出不可用。
  - 避免把所有文件下载都塞进 JSON API 假设里。
- **风险**：
  - 如果采用 signed URL，需要额外管理短期 token 的签发和过期。
  - 如果扩展 secure 网关，需要明确响应体大小限制。
- **验证**：
  - `FORCE_SECURE_API=true` 下导出 CSV 成功。
  - 普通用户无法导出管理员日志。
  - 大小在预期范围内的 CSV 文件不会被 JSON 解析破坏。

## 选项 D：暂不处理

- **做法**：维持现状，仅记录风险。
- **收益**：无改动成本。
- **风险**：
  - 会话、缓存和权限问题继续累积。
  - 后续再修时需要同时处理更多历史状态和兼容问题。
- **验证**：N/A

---

## 推荐

推荐分两步执行：

1. 先采用选项 A，做低风险补丁，解决确定性运行时错误和权限边界缺口。
2. 随后采用选项 B，作为一次独立 DEC 管理的认证会话重构。

下载链路选项 C 可以独立排期，不应阻塞认证会话主链路。

**理由**：

当前项目是个人项目，没必要一次性做成企业级安全工程，但认证会话链路已经影响到实际功能稳定性。先补明显 bug，再拆缓存职责和会话身份模型，成本更可控。

---

## 前置条件 / 依赖

1. 明确是否允许旧 token 全部失效，若允许，认证重构可简化。
2. 明确多设备登录是否仍是目标能力；如果不是，可以将策略收敛为单设备登录。
3. 在修复前保留当前 `main` 分支状态，避免与已有未提交文档和部署脚本变更混淆。
4. 修复后至少补充一组认证/会话接口级测试，覆盖 login、refresh、logout、revoke。
