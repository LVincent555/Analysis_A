# DEC-001: 文档迁移与 PSDR 流程统一

**日期**: 2026-07-07
**状态**: [ACTIVE]
**类型**: Infra / Documentation / Governance
**层级**: 顶层
**关联**: 无

---

## 决策

`stock_analysis_app` 的项目文档治理主入口迁移到 `PSDR-Review/`。

自本决策生效起，后续所有与 `stock_analysis_app` 相关的正式问题记录、方案建议、决策记录、执行结果、复盘记录和长期有效文档，统一进入 `PSDR-Review` 流程管理：

- `PRB/`: 问题记录，描述现象、影响、证据和待澄清点。
- `SUG/`: 方案建议，提出可选路线、风险、成本和执行步骤。
- `DEC/`: 决策记录，确认采用路线、边界、负责人和验收标准。
- `RES/`: 结果复盘，记录执行结果、验证证据、遗留风险和后续动作。
- `META/`: 流程规范、索引和长期有效的治理说明。

原 `docs/` 目录中的旧文档记录统一视为历史资料，封存归档，作为查证、追溯和迁移参考使用，不再作为新增正式文档的主入口。

---

## 背景

`stock_analysis_app` 已经积累了大量功能设计、部署、优化、测试、问题修复和历史版本文档。旧文档集中在 `docs/` 目录及其 `_archive/` 子目录中，内容价值较高，但存在以下治理问题：

1. 文档类型混杂，设计、决策、结果、操作手册和历史记录边界不清。
2. 新旧文档共存，后续维护时不容易判断哪份仍然有效。
3. 决策链路缺少统一编号和状态，难以追踪一个问题从提出到落地的全过程。
4. `docs/` 已经承担历史仓库角色，继续向其中追加正式文档会扩大混乱。

因此需要建立新的项目文档主线，并将旧文档封存为历史记录。

---

## 范围

**做**：
- 将 `PSDR-Review/` 作为 `stock_analysis_app` 后续正式文档和决策流程的主目录。
- 所有新增正式文档按 `PRB -> SUG -> DEC -> RES` 的链路组织。
- 所有新增决策必须进入 `PSDR-Review/DEC/`，从 `DEC-001` 起编号递增。
- 所有新增执行结果和复盘必须进入 `PSDR-Review/RES/`。
- 旧 `docs/` 目录保留为历史资料和迁移来源，原则上只读封存。
- 如旧文档仍有长期有效内容，应以新的 PSDR 文档形式摘录、重述或迁移，不直接在旧文档上继续扩展。

**不做**：
- 不在本决策中删除旧 `docs/` 文档。
- 不立即强制搬迁所有历史文档，以免破坏已有链接、部署说明或外部引用。
- 不把临时草稿、运行日志、缓存文件、构建产物放入 `PSDR-Review/`。
- 不用 `docs/` 承载新的正式决策链路。

---

## 文档归档规则

1. `docs/` 保留为历史归档区。
2. `docs/_archive/` 中已有历史资料继续保持归档状态。
3. `docs/` 根目录中仍有参考价值的旧文档，后续可按主题迁移为 `PSDR-Review/PRB|SUG|DEC|RES` 中的新文档。
4. 迁移不是简单复制，应补齐状态、日期、关联关系、验收标准和结果证据。
5. 若旧文档只用于查证历史，不需要迁移，只在新文档中引用其路径即可。

---

## 新增文档规则

从本决策生效起：

1. 发现问题时，先写 `PRB-XXX`。
2. 提方案时，写 `SUG-XXX`，并关联对应 `PRB`。
3. 形成选择时，写 `DEC-XXX`，并说明采纳范围和不采纳范围。
4. 执行完成或阶段结束时，写 `RES-XXX`，记录验证结果和遗留风险。
5. 所有正式编号文档必须同步更新 `PSDR-Review/META/INDEX.md`。

轻量说明、一次性草稿、临时排查记录可以不进入 PSDR；一旦内容会影响后续开发、部署、运维、数据安全或产品方向，就必须进入 PSDR。

## 验收标准

1. `PSDR-Review/DEC/DEC-001-文档迁移与PSDR流程统一.md` 已创建。
2. `PSDR-Review/META/INDEX.md` 已登记本决策。
3. 后续 `stock_analysis_app` 新增正式文档不再写入 `docs/` 作为主入口。
4. 旧 `docs/` 文档不被删除，作为历史归档保留。
