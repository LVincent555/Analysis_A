# PRB-003: 后端架构迁移双轨与依赖漂移风险

**日期**: 2026-07-08
**状态**: [OPEN]
**类型**: Infra / Backend / Architecture / Migration / Risk
**层级**: Backend
**关联**: SUG-003, DEC-002, RES-004

---

## 1. 现象

后端准备从 Java 式三层目录迁移到“模块化单体 + 六边形架构 + 局部 DDD + CQRS 查询侧”。迁移期间不可避免会出现新旧目录并存：

- 旧目录：`routers/`、`services/`、`auth/`、`db_models.py`
- 新目录：`contexts/{identity,market_data,analysis,board_heat,operations}/...`

如果没有明确的迁移边界、兼容壳和测试基线，容易形成“新目录下的新混乱”，甚至比当前三层结构更难维护。

---

## 2. 主要风险

1. 新旧目录长期并存，新增功能继续写入旧 `routers/` / `services/`，迁移无法收敛。
2. FastAPI router 路径、响应字段、状态码在迁移中发生隐性变化，前端和 Electron 客户端被动破坏。
3. `main.py` 同时承担路由注册、缓存初始化、同步器、审计回调，composition root 不清晰。
4. `db_models.py` 混合股票、板块、用户、会话、配置、日志、角色模型，直接拆分会影响 SQLAlchemy metadata 和大量 import。
5. `secure.py`、`auth_middleware.py`、`crypto/`、`core/caching/` 属于横切平台能力，归属不清会导致 Identity、Operations、Analysis 互相穿透。
6. `admin.py` 同时包含后台运维、数据导入、数据删除和登录历史查询，不应整体归入 Operations。
7. 查询侧接口大量依赖缓存、Numpy/Pandas、日期格式和持久化读模型，迁移时容易造成缓存 key 重影或结果不一致。
8. 现有测试混合单元测试、脚本式 API 测试、真实数据测试和真实服务测试，默认测试集合不稳定。
9. `backend/app/database.py` 被 `.gitignore` 忽略，仓库只追踪 `database.py.example`；fresh clone、测试基线和真实运行之间存在配置文件漂移风险。

---

## 3. 影响

严重度：高。

影响范围：

- 后端架构迁移节奏
- 登录会话与权限边界
- 分析查询和板块热度查询一致性
- 缓存失效和预热行为
- 前端 API 兼容性
- 后续新增功能落点

可能出现的问题：

1. 重构期间旧功能回归困难。
2. 新旧 API 行为不一致，但缺少自动化测试及时发现。
3. 某些领域规则从旧 router 搬到新 use case 时被遗漏。
4. 分析查询迁移后性能下降或缓存命中失效。
5. 后端架构文档看似完整，但实施者缺少文件级清单，执行时分叉。

---

## 4. 初步约束

1. 架构迁移必须先建立 `uv` + pytest 稳定测试基线。
2. 默认测试只运行离线单元测试，真实服务/真实数据/外部网络测试必须进入 integration profile。
3. 每个 Level 迁移前冻结对应 API contract，迁移后对比路径、状态码和 JSON key。
4. 旧路径必须通过兼容壳保留，除非另立 DEC 批准破坏性变更。
5. ORM 物理拆分不能作为第一步；先建立上下文归属表和 repository adapter。
6. `domain` import 边界后续应加入自动检查。
7. Level 0 需要决定数据库连接模块的版本化方式，避免业务代码依赖不可追踪的本地文件。

---

## 5. 下一步

1. 将本风险纳入 `SUG-003` 和 `DEC-002` 的实施约束。
2. 建立 `RES-004` 记录当前 `uv` 测试基线落地结果。
3. 在 Level 0 补齐 composition root、API contract、ORM ownership、平台能力归属说明。
4. 在 Level 0 处理 `database.py` / `.env` / `database.py.example` 的配置治理。
5. 后续每个 Level 完成后单独写 RES，不提前关闭本 PRB。
