# PRB-001: 后端认证会话与缓存职责混用风险

**日期**: 2026-07-08
**状态**: [OPEN]
**类型**: Infra / Backend / Auth / Cache / Audit
**层级**: Backend
**关联**: SUG-001, RES-001

---

## 1. 现象

本次只读审查发现，后端认证、会话、加密网关和统一缓存之间存在职责边界混用，主要事实如下：

1. `sessions` 缓存分区在 `backend/app/main.py` 中注册为 `WriteBehindPolicy(ttl=0)`，名称和策略看起来面向“会话心跳/会话状态”。
2. `backend/app/auth/dependencies.py` 的 `set_session_key()`、`get_session_key_by_user_id()`、`remove_session_key()` 使用同一个 `sessions` 分区，按 `user_id` 存取 `/api/secure` 解密所需的 `session_key`。
3. `backend/app/core/caching/facade.py` 的 `set_session_heartbeat()` 也设计为使用 `sessions` 分区，但按 `session_id` 写入心跳状态、最后活跃时间和 IP。
4. `backend/app/core/caching/syncer.py` 的 `_sync_sessions()` 会把 `sessions` 分区的 dirty key 解释为 `UserSession.id`，并批量写入 `user_sessions` 表。
5. `backend/app/auth/dependencies.py:get_current_user()` 只校验 JWT、用户存在和 `is_active`，未校验 `UserSession.is_revoked`、`UserSession.expires_at`、`User.token_version`、账号 `deleted_at`、账号 `expires_at`。
6. `backend/app/services/session_service.py` 撤销会话时递增 `User.token_version`，但 JWT 生成和校验链路未写入或读取 `token_version`。
7. `backend/app/routers/auth.py` 的 refresh 失败路径调用 `logger.warning()`，但文件未定义 `logger`。
8. `backend/app/routers/auth.py:logout()` 只删除内存 session key，不撤销或删除数据库中的当前 `UserSession`。
9. `backend/app/routers/cache_mgmt.py` 的 `/api/cache/*` 管理接口没有路由级管理员依赖。
10. `backend/app/routers/log_mgmt.py` 的 CSV 导出返回 `StreamingResponse`，但 `backend/app/routers/secure.py` 的加密网关固定按 JSON 解析内部响应。

## 2. 影响

严重度：高。影响范围集中在登录态、加密 API、管理员会话管理、缓存管理和日志导出。

可能出现的用户可见问题：

1. 多设备登录时，后登录设备覆盖同一用户的内存 `session_key`，旧设备继续访问 `/api/secure` 时可能解密失败或表现为会话异常。
2. `sessions` 缓存 dirty 数据被同步器误认为 `UserSession.id`，可能错误更新 `user_sessions` 表中的 `last_active`、`current_status`、`ip_address`。
3. 管理员“强制下线”“撤销会话”不一定能立即阻断已签发的 access token。
4. refresh token 异常路径可能返回 500，而不是预期的 401。
5. 普通登录用户可能通过 secure 网关调用缓存管理接口，触发清缓存、重载缓存或 GC。
6. 操作日志 CSV 导出在强制 secure 模式下存在前后端调用方式不兼容。

## 3. 初步假设

1. 早期实现将“运行时解密 key 缓存”和“会话心跳 write-behind 缓存”都归入 `sessions`，后续演进没有拆分分区。
2. 会话表、JWT、加密网关之间缺少统一的“当前会话身份”概念，导致 token 只携带 `user_id/device_id`，认证依赖无法准确绑定 DB 会话记录。
3. `token_version` 是后来为强制下线设计的字段，但 JWT payload 和认证依赖没有同步演进。
4. 管理类接口主要依赖全局认证中间件和 secure 网关，部分路由遗漏了细粒度管理员权限依赖。
5. 加密网关最初面向 JSON API 设计，未覆盖文件下载、流式响应等非 JSON 响应形态。

## 4. 下一步

需要进一步验证：

1. 构造两个不同 `device_id` 登录同一用户，确认后登录设备是否覆盖前一设备的 `session_key`。
2. 登录后观察 `UnifiedCache.session()` dirty keys，确认同步器是否会把 `user_id` 当 `UserSession.id` 写库。
3. 撤销某个会话后，使用旧 access token 调用普通业务 API 和 `/api/secure`，确认是否仍可访问。
4. 使用失效或不存在的 refresh token 调 `/api/auth/refresh`，确认是否触发 `NameError`。
5. 使用普通用户调用 `/api/cache/clear|reload|gc`，确认权限边界。
6. 在 `FORCE_SECURE_API=true` 下测试操作日志 CSV 导出链路。
